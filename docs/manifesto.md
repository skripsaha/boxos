МАНИФЕСТ АРХИТЕКТУРЫ BOXOS (SNOWBALL)

ВАЖНО: BoxOS — это НЕ event-driven архитектура. Структура данных называется "Event", но это command packet (пакет команд). В BoxOS нет подписок на события, нет callbacks, нет pub/sub паттерна. Процесс явно отправляет команду (notify) и явно читает ответ (Result Page). Архитектура BoxOS — Snowball Command Pipeline: команда катится через цепочку обработчиков (Деков).

1. ФИЛОСОФИЯ: ОТКАЗ ОТ ТРЕНИЯ
   Современные ОС перегружены "трением": лишними переключениями контекста, сложными деревьями папок и тысячами системных вызовов. BoxOS уничтожает трение, заменяя хаос упорядоченным «Снежным комом» данных.

2. ГЛАВНЫЙ ШЛЮЗ: ИНТЕРФЕЙС notify()
   В системе нет таблицы системных вызовов. Существует только одна точка входа в ядро — INT 0x80.

Механизм: Процесс заполняет Notify Page (0x1000, 8KB) цепочкой префиксов (до 16) и данными (до 256 байт), после чего вызывает notify().

Обработка: Ядро копирует запрос в EventRing, ставит процесс в WAITING, и Guide обрабатывает событие через цепочку Деков.

Результат: Доставляется в Result Page (0x3000, 28KB) — кольцевой буфер из 111 слотов. Процесс получает CPU обратно с бонусом Hot Result (+50) от планировщика.

3. ЯДРО: ЦЕПОЧКА «СНЕЖНОГО КОМА»
   Ядро BoxOS — это не библиотека функций, а активный маршрутизатор.

Центральные компоненты:

Guide (Маршрутизатор): Batch-обработчик, который берет событие из EventRing и прогоняет его по цепочке Деков. Вызывается из планировщика.

Decks (Функциональные блоки):

ID 01: Operations — Трансформация данных.

ID 02: Storage — Логика Tagfs.

ID 03: Hardware — Драйверы и железо.

ID FF: System — Права, процессы и маппинг.

Execution Deck: Финализатор. Доставляет результат в Result Page и устанавливает result_there для планировщика.

4. ФАЙЛОВАЯ СИСТЕМА: TAGFS
   Мы навсегда уходим от иерархии папок. Данные — это объекты в семантическом облаке.

Обычные теги (key:value): Организация (напр. type:code, project:os).

Системные теги (Label): Права и роли (напр. app, utility, system).

Поиск: Мгновенное пересечение множеств тегов. Нет путей — есть только свойства.

5. КИЛЛЕР-ФИЧА: КОМАНДА use И КОНТЕКСТ
   Контекст в BoxOS — это не просто строка в терминале, это физическое изменение приоритетов системы.

Команда: use project:mars

Эффект: Система скрывает всё лишнее. Планировщик видит теги процесса и, если они совпадают с текущим use, дает процессу +20 баллов приоритета.

Наследование: Теги контекста передаются по наследству, создавая "рабочие зоны" на уровне железа.

6. ПЛАНИРОВЩИК: SMART SCORE (БАЛЛЬНАЯ СИСТЕМА)
   Процессор получает тот, кто максимально полезен здесь и сейчас:

+50 баллов: Только что пришел результат из ядра (Hot Data, затухает).

+20 баллов: Процесс соответствует контексту use.

+10/+30/+100 баллов: Три уровня защиты от голодания (Mild/Severe/Critical).

+200 баллов: Восстановление после переполнения Result Ring.

-100 баллов: Штраф за 5+ подряд запусков (Consecutive Run Penalty).

7. ЗОЛОТОЕ ПРАВИЛО ИСПОЛНЕНИЯ
   Мы объединяем безопасность ядра и скорость "голого железа":

State-Changing: Любое действие, влияющее на систему (запись на диск, сеть), идет через цепочку Деков и проверку Security Gate.

Non-State-Changing: Игры и тяжелые вычисления получают права через System Deck и работают напрямую с выделенными буферами (Kernel Bypass), минуя ядро.

ВЕРДИКТ И ИТОГИ

BoxOS (Snowball) — это первая в мире семантическая операционная система. Она не заставляет пользователя подстраиваться под структуру папок и прерывания процессора. Напротив, она подстраивает ресурсы процессора и видимость данных под намерения пользователя.

Ни в одной ОС планировщик не привязан к семантическим тегам (use).

Ни в одной ОС нет такой плотной интеграции "Снежного кома", где данные мутируют внутри ядра без лишних переключений контекста.

Это модульная, быстрая и невероятно логичная система.
