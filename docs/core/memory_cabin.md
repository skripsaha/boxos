Концепция Cabin (Каюта): Изоляция и Управление Памятью
Cabin — это изолированное виртуальное адресное пространство, выделенное процессу. Это «личная комната» программы, границы которой охраняются аппаратными средствами процессора (MMU). Каждый Cabin имеет собственный CR3 (PML4 page table).

1. Структура Каюты
   Каждая Каюта имеет фиксированную внутреннюю топологию. Для процесса внутри мир всегда выглядит одинаково:

```
Виртуальный адрес          Размер    Назначение
─────────────────────      ────────  ────────────────────────────
0x0000 - 0x0FFF            4KB       NULL Trap Zone (не маппится)
0x1000 - 0x2FFF            8KB       Notify Page (User → Kernel, 2 физических страницы)
0x3000 - 0xBFFF            36KB      Result Page (Kernel → User, 9 физических страниц)
0xC000 - ...               var       Code + Data + BSS (entry point, flat binary)
0x20000 - ...              var       Heap (растёт вверх)
0x40000000 - ...           var       Buffer Heap (маппинг ядерных буферов)
0x7FFFF000                 4KB       CPU Caps Page (read-only)
0x7FFFFFFFE000             var       User Stack (растёт вниз, 64KB + 1 guard page)
```

Определяющий файл: `src/include/cabin_layout.h` — единственный источник правды для всех адресов.

2. Аппаратная изоляция
   Ядро через VMM создаёт для каждого процесса собственный PML4 (4-уровневые page tables x86-64). Cabin context включает:
   - Собственные PML4[0] (low memory) с копией kernel identity mapping + user pages
   - Общие PML4[256-511] с kernel (higher-half kernel space)

Процесс физически не может сгенерировать адрес, выходящий за пределы своей Каюты. Любая попытка приводит к Page Fault → уничтожению процесса → освобождению ресурсов.

3. Управление памятью: Страничный механизм
   Несмотря на то что для процесса память кажется «непрерывной», на физическом уровне ядро собирает Каюту из разрозненных 4KB страниц через PMM (Physical Memory Manager):

- PMM выделяет физические страницы (bitmap allocator)
- VMM маппит их на фиксированные виртуальные адреса Cabin
- Для программы это «литая каюта», а для ядра — набор кирпичиков

Это полностью решает проблему внешней фрагментации — физическая непрерывность не требуется.

4. Жизненный цикл

Бронирование: При PROC_SPAWN, vmm_create_cabin() создаёт PML4 и выделяет физические страницы для Notify (2 pages) и Result (9 pages).

Заселение: Flat binary копируется в Каюту начиная с 0xC000. BSS обнуляется. Stack маппится на 0x7FFFFFFFE000.

Эксплуатация: Процесс живет внутри, общаясь с ядром через Notify Page (0x1000) и Result Page (0x3000).

Санация: При завершении процесса ядро освобождает все физические страницы через VMM/PMM и уничтожает PML4.

5. Kernel Stack
   Каждый процесс имеет отдельный kernel stack для обработки прерываний:
- Размер: 4 pages (16KB) + 1 guard page
- Выделяется динамически через pmm_alloc() при создании процесса
- TSS.RSP0 указывает на вершину kernel stack (переключается в context_restore)
- При timer interrupt (INT 0x20), CPU автоматически переключается на kernel stack через TSS
