Концепция Cabin (Каюта): Изоляция и Управление Памятью
Cabin — это изолированная, непрерывная область физической памяти, выделенная процессу. Это «личная комната» программы, границы которой охраняются аппаратными средствами процессора (MMU).

1. Структура Каюты
   Каждая Каюта имеет фиксированную внутреннюю топологию. Для процесса внутри мир всегда выглядит одинаково:

0x0000 - 0x0FFF: Зарезервировано (Trap-зона для нулевых указателей).

0x1000 - 0x1FFF: Notify Page (Вход в ядро).

0x2000 - 0x2FFF: Result Page (Выход из ядра).

0x3000 - [End]: Полезная нагрузка (Код, статические данные, куча и стек).

2. Аппаратная изоляция
   Ядро (через System Deck) настраивает таблицы страниц так, что процесс физически не может сгенерировать адрес, выходящий за пределы его Каюты. Любая попытка «выйти в коридор» без использования notify() приводит к мгновенному уничтожению процесса (Segmentation Fault) и освобождению Каюты.

3. Проблема фрагментации и её решение в BoxOS
   Поскольку Каюта — это непрерывный блок, при частом запуске и остановке программ память может стать похожа на «сыр с дырками» (внешняя фрагментация). У нас есть три уровня решения этой проблемы:

А. Страничный механизм (Paging)

Несмотря на то что для процесса память кажется «непрерывной», на физическом уровне ядро может собирать Каюту из разрозненных страниц по 4 КБ.

Инновация: Для программы это «литая каюта», а для ядра — набор кирпичиков. Это полностью решает проблему поиска больших кусков свободной памяти.

Б. Метод «Схлопывания» (Defragmentation)

Поскольку наши процессы общаются с миром только через notify(), они асинхронны. Это позволяет ядру в моменты простоя (idle) физически перемещать Каюты в памяти, обновляя таблицы страниц.

Процесс даже не заметит, что его «каюту» перенесли с одной палубы на другую, так как его виртуальные адреса (0x3000...) не изменятся.

В. Фиксированные размеры (Standard Cabins)

Чтобы минимизировать дробление, System Deck оперирует стандартными размерами кают (например: 64КБ, 1МБ, 16МБ).

Если приложению нужно 100КБ, оно получает каюту в 128КБ. Это создает небольшую внутреннюю фрагментацию, но делает внешнюю фрагментацию предсказуемой и легкой для управления.

4. Жизненный цикл
   Бронирование: При наличии тега app, System Deck выделяет место.

Заселение: Код копируется в Каюту.

Эксплуатация: Процесс живет внутри, общаясь с Деками через страницы 0x1000/0x2000.

Санация: При выходе из процесса ядро обнуляет Каюту (защита данных) и помечает блок как свободный.

Вердикт по фрагментации:

В BoxOS фрагментация — это не приговор, а управляемый параметр. Благодаря тому, что у нас нет прямых указателей на физическую память в Userspace, ядро остается абсолютным хозяином пространства.
