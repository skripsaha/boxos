Название: Анатомия процесса и интерфейс Userspace Тип: Спецификация программного окружения (Runtime Environment) Базовый принцип: Изоляция через Арены и коммуникация через фиксированные Shared-страницы.

1. Карта памяти процесса (Virtual Memory Map)
   Каждый процесс в BoxOS видит стандартную топологию своего виртуального адресного пространства. Это упрощает компиляцию и делает работу с памятью предсказуемой.

Адрес (Virtual) Название страницы Доступ (User) Описание
0x0000 - 0x0FFF Reserved No Access Защита от разыменования NULL-указателя.
0x1000 Notify Page Read/Write Шлюз для отправки команд в ядро.
0x2000 Result Page Read-Only Кольцевой буфер ответов от ядра.
0x3000 - 0x7FFF Static Code/Data R / RW Тело программы и глобальные переменные.
0x8000+ Dynamic Arenas RW Динамическая память процесса (Hardware-backed). 2. Структура Notify Page (0x1000)
Эта страница — «бланк заказа», который процесс заполняет перед вызовом notify().

C
struct NotifyPage {
uint32_t magic; // Сигнатура (например, 0xBAADF00D)
uint16_t prefixes[128]; // Цепочка инструкций (Snowball Pipeline)
uint8_t data[3836]; // Полезная нагрузка (Payload)
};
Как данные попадают в Notify Page:

Формирование цепочки: Runtime-библиотека или компилятор записывает последовательность префиксов в массив prefixes. Конец цепочки всегда 0x0000.

Упаковка данных: Аргументы функций (например, ID файла, смещение, сетевой адрес) упаковываются в буфер data.

Установка Magic: В последнюю очередь пишется magic, подтверждающий готовность страницы.

Вызов: Процесс исполняет инструкцию прерывания, триггерящую функцию ядра notify().

3. Структура Result Page (0x2000)
   Страница, куда ядро (через Execution Deck) доставляет результаты. Процесс может опрашивать её (polling) или ждать прерывания.

C
struct ResultPage {
uint32_t magic; // Валидация страницы ядра
struct ResultRing {
uint32_t head; // Индекс чтения (управляет процесс)
uint32_t tail; // Индекс записи (управляет ядро)
struct ResultEntry {
uint16_t status; // Код успеха/ошибки
uint16_t size; // Размер вернувшихся данных
uint8_t payload[252];
} entries[15]; // Кольцо из 15 результатов
} ring;
}; 4. Жизненный цикл "Запрос-Ответ" (Userspace Side)
Подготовка: Приложение вызывает библиотечную функцию, например storage_read("type:config").

Заполнение 0x1000: Библиотека пишет префикс 02:01 (Storage Query) в prefixes[0] и теги в data.

Активация: Выполняется notify(). Процесс может продолжать другие вычисления.

Ожидание (Optional): Если приложению нужен результат немедленно, оно проверяет ResultRing.tail.

Получение: Как только tail изменился, приложение забирает данные из payload и сдвигает свой head.

5. Взаимодействие с Аренами (Hardware Deck)
   Процесс не управляет памятью через malloc.

Если приложению нужно больше места, оно отправляет префикс в System Deck (0xFF): FF:02 (MEM_ARENA_GRANT).

Ядро аппаратно расширяет доступный диапазон страниц в MMU.

Для процесса это выглядит как внезапное появление доступных адресов выше 0x8000.

6. Контекст исполнения и Теги
   В PCB (внутри ядра) для каждого процесса хранится его Tag List.

При наследовании: Если parent_process имел тег project:secret, то child_process при создании через PROC_SPAWN автоматически получит этот тег.

При выполнении: Любая попытка доступа к файлам или сети будет неявно фильтроваться планировщиком и ФС на основе этих тегов.

Итог: Почему это удобно для программиста?

Никаких скрытых состояний: Всё общение с ОС происходит через две страницы памяти. Это делает отладку невероятно простой — достаточно посмотреть дамп памяти по адресу 0x1000 и 0x2000.

Безопасность «из коробки»: Процесс физически не может отправить в ядро ничего, кроме того, что влезет в Notify Page.

Асинхронность как стандарт: Программист привыкает писать код, который не ждет ответа от диска, а продолжает работать, что делает приложения в BoxOS очень отзывчивыми.
