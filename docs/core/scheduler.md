Название: Smart Score Scheduler (Семантический планировщик) Тип: Балльно-приоритетная система распределения ресурсов CPU. Главный принцип: Фокус на пользователе и актуальности данных (Hot Data).

1. Философия планирования
В отличие от классических ОС, которые используют сложные формулы квантования времени, Scheduler в BoxOS опирается на Баллы Приоритета (Scores). Система всегда отдает квант времени тому процессу, у которого в данный момент высший балл.

2. Система начисления баллов (The Scoring System)
Баллы динамически пересчитываются при каждом событии в системе.

Категория	Баллы	Триггер (Условие)
Hot Result	+50	Процесс только что получил данные из Execution Deck.
Context Match	+20	Теги процесса полностью или частично совпадают с текущим глобальным use контекстом.
Starvation Fix	+10	Процесс не получал время CPU более 100мс (Watchdog).
System Utility	+5	Процесс имеет системный тег utility или system.
Обоснование баллов:

Hot Result (+50): Самый высокий приоритет. Если данные только что вышли из ядра, они еще лежат в кэше процессора (L1/L2). Нужно дать процессу поработать с ними немедленно, пока они «горячие».

Context Match (+20): Если пользователь ввел use project:mars, то все процессы, помеченные этим тегом, становятся важнее остальных. Система «понимает», над чем работает человек.

Starvation Fix (+10): Защита от «голодания». Если низкоприоритетный процесс долго ждет, он постепенно накапливает баллы, пока не получит свой квант времени.

3. Механика Контекста (use) и Наследования
Scheduler тесно связан с командой use, которая хранится в System Deck.

Глобальный контекст: Пользователь или система устанавливает активный набор тегов.

Сверка тегов: При выборе процесса планировщик сравнивает теги в PCB (Process Control Block) с активными тегами use.

Наследование: * При spawn() дочерний процесс наследует теги родителя.

Это гарантирует, что если ты запустил компилятор внутри «рабочего контекста», все его подпроцессы тоже будут в приоритете.

4. Алгоритм выбора процесса (The Selection Loop)
Планировщик работает в связке с EventRing.

Check Results: Scheduler проверяет флаги result_there в PCB всех активных процессов. Те, у кого флаг поднят, получают +50 и перемещаются в топ очереди.

Filter Context: Всем процессам, чьи теги соответствуют активному use, добавляется +20.

Sort & Run: Выбирается процесс с максимальным суммарным баллом.

Quantum: Процессу выделяется фиксированный или динамический квант времени.

Decay (Затухание): После выполнения кванта бонусные баллы (кроме контекстных) сгорают или уменьшаются, чтобы дать шанс другим.

5. Оптимизация Idle (Энергоэффективность)
Если EventRing пуст и ни у одного процесса нет баллов выше базового порога:

Scheduler отправляет ядро в состояние Deep Sleep (инструкция HLT).

Пробуждение происходит только по аппаратному прерыванию (таймер или ввод-вывод) или через notify().

6. Псевдокод логики (Core Logic)
C
Process* scheduler_select_next() {
    Process* best_candidate = NULL;
    int max_score = -1;

    for (each Process p in ProcessList) {
        int current_score = 0;

        // 1. Бонус за свежий результат (Hot Data)
        if (p->result_there) current_score += 50;

        // 2. Бонус за совпадение с 'use' контекстом
        if (match_tags(p->tags, current_system_context)) current_score += 20;

        // 3. Бонус за время ожидания (Watchdog)
        current_score += (current_time - p->last_run_time) / STARVE_RATIO;

        if (current_score > max_score) {
            max_score = current_score;
            best_candidate = p;
        }
    }

    return best_candidate;
}
Почему это инновация?

В классических ОС планировщик «слеп». Он не знает, что chrome.exe сейчас важен, а background_update.exe — нет, пока один не начнет тормозить другой.

В BoxOS планировщик — это часть интерфейса пользователя. Командой use ты буквально перенастраиваешь физику своего процессора под конкретную задачу. Это и есть «понимающая» ОС.
