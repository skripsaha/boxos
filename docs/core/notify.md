Название: notify() — Универсальный системный шлюз Тип: Единственный системный вызов (Universal Entry Point) Механизм: Переход из Ring 3 в Ring 0 для инициации «Снежного кома».

1. Концепция
   В BoxOS отсутствует таблица системных вызовов (syscall table). Вместо сотен функций существует только одна инструкция — notify(). Она не передает параметры через регистры в классическом понимании, а служит сигналом ядру проверить общую страницу памяти процесса (Notify Page) и сформировать событие.

2. Предварительные условия (Userspace)
   Перед вызовом notify(), процесс обязан подготовить данные на странице Notify Page (0x1000 virtual):

Magic: Записать сигнатуру валидности.

Prefixes Array: Сформировать массив 16-битных команд (цепочка маршрута).

Data: Поместить необходимые аргументы или payload в буфер данных.

3. Алгоритм работы notify() (Kernel Side)
   При срабатывании прерывания/инструкции notify(), ядро выполняет следующие шаги строго по порядку:

Save Context: Сохранение состояния регистров текущего процесса в стек ядра.

Privilege Transition: Переход процессора в Ring 0.

Identify: Получение current_pid исполняемого процесса.

PCB Lookup: Поиск структуры PCB (Process Control Block) в системной таблице по PID.

Address Resolution: Получение физического адреса страницы памяти процесса, отображенной на виртуальный адрес 0x1000 (Notify Page).

Validation: Проверка magic числа на странице. Если magic неверный — мгновенный возврат с ошибкой.

Event Creation: Аллокация новой структуры Event во внутренней памяти ядра.

Prefix Ingestion:

memcpy() массива префиксов из Notify Page в event->prefixes[].

Важно: Ядро принудительно добавляет 0x0000 (терминатор) в конец массива для безопасности.

Data Ingestion: memcpy() буфера данных из Notify Page в event->data.

Event Initialization:

event->pid = current_pid;

event->state = NEW;

event->current_prefix_idx = 0;

Обнуление массива результатов event->results[].

Guide Awakening:

Проверка состояния EventRing.

Если буфер был пуст: Guide.state = READY; (пробуждение центрального диспетчера).

Queuing: Помещение сформированной структуры Event в хвост EventRing.

Asynchronous Return:

Восстановление регистров процесса из стека.

Выполнение инструкции iret (или аналогичной) для возврата в Userspace.

Результат: Процесс продолжает работу сразу, не дожидаясь исполнения цепочки (Асинхронность).

4. Структура взаимодействия
   Plaintext
   [ Userspace Process ]
   |
   | 1. Заполняет Notify Page (0x1000)
   | 2. Вызывает notify()
   V
   [ KERNEL: notify_gateway ]
   |
   | 3. Сохраняет регистры
   | 4. Копирует Prefixes + Data -> Event
   | 5. Кладет Event в EventRing
   | 6. Будит Guide
   | 7. iret (Возврат)
   V
   [ Userspace Process ] ----> (Продолжает вычисления)
5. Технические особенности
   Асинхронность: notify() — это выстрел "выстрелил и забыл". Процесс узнает о результате позже через Result Page (0x2000).

Безопасность: Ядро копирует данные (memcpy), а не работает с ними прямо на странице пользователя. Это защищает ядро от попыток процесса изменить префиксы "на лету" (Time-of-check to time-of-use attack).

Производительность: Благодаря тому, что notify() один, логика входа в ядро максимально оптимизирована и может быть вписана в несколько десятков тактов процессора.
