Trigger: Prefix 0x0000 (терминатор цепочки префиксов)

Название: Execution Deck (Финализатор)

Роль: Завершение жизненного цикла события, доставка результата и пробуждение процесса.

ВАЖНО: Execution Deck НЕ является обычным Deck'ом и НЕ регистрируется в deck_table.
Он вызывается Guide напрямую при обнаружении префикса 0x0000 или при ошибках в цепочке префиксов.

Описание

Execution Deck вызывается автоматически диспетчером Guide, когда тот встречает в массиве префиксов терминатор 0x0000. Этот Дек является мостом между внутренним command pipeline ядра и контекстом исполнения пользователя.

Алгоритм работы (The Delivery Logic)

Идентификация: Извлекает event->pid для поиска управляющей структуры процесса (PCB).

Доступ к результату: Находит физический адрес Result Page (0x3000 в вирт. пространстве процесса, 36KB).

Запись в Ring: * Проверяет свободное место в ResultRing.

Копирует итоговое содержимое event->data в хвост (Tail) кольцевого буфера.

Записывает метаданные: статус выполнения (OK/ERROR) и размер данных.

Синхронизация: * Сдвигает указатель Tail в Result Page.

Устанавливает в PCB флаг result_there = true.

Cleanup (Утилизация): * Освобождает память, занятую структурой Event в ядре (kfree).

Обнуляет временные буферы, использованные в цепочке.

Scheduler Signal: * Отправляет сигнал планировщику: «Процесс PID готов к работе».

Начисляет процессу +50 баллов приоритета (Hot Result), так как данные уже в кэше процессора.

Таблица Opcodes

У Execution Deck нет множества опкодов, так как он выполняет одну фундаментальную задачу. Однако, в зависимости от состояния event->state, он ведет себя по-разному:

Состояние Event	Действие	Результат в ResultRing
STATE_SUCCESS	Стандартная доставка.	Код 0 + Данные.
STATE_ERROR	Доставка отчета об ошибке.	Код ошибки + Дамп data.
STATE_CRITICAL	Экстренное завершение.	Код краша, уведомление System Deck.
Почему это важно для производительности?

Мгновенный возврат: Благодаря тому, что результат кладется в Result Page (shared memory), приложению не нужно делать системный вызов read(), чтобы узнать ответ. Оно просто видит обновление в своем кольцевом буфере.

Автоматическая чистота: Разработчику приложения не нужно заботиться об удалении структур событий. Как только «снежный ком» докатился до Execution Deck, ядро само подметает за ним весь мусор.

Приоритет «по горячим следам»: Поскольку Execution Deck сообщает планировщику о завершении, процесс просыпается именно тогда, когда данные максимально «свежие» в кэше L3.
