РАБОТА С GIT В ПРОЕКТЕ BOXOS

Этот документ описывает соглашения и рабочий процесс git для разработчиков BoxOS.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. БЫСТРЫЙ СТАРТ (TL;DR)

   git add src/kernel/core/guide/guide.c       # добавить конкретный файл
   git commit -m "fix(guide): исправить обработку пустого EventRing"
   git push origin main

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

2. СТРУКТУРА COMMIT-СООБЩЕНИЯ

   Формат:

       <тип>(<область>): <краткое описание>

       [необязательное тело: что и почему, не как]

   Тип — одно из:

       feat     — новая функциональность
       fix      — исправление бага
       refactor — рефакторинг без изменения поведения
       docs     — изменения в документации
       test     — добавление или правка тестов
       build    — изменения в Makefile / системе сборки
       chore    — прочее (переименование, очистка мусора)

   Область — подсистема, которую затронул коммит:

       guide       — маршрутизатор Guide
       scheduler   — планировщик
       pmm / vmm   — физическая / виртуальная память
       tagfs       — файловая система тегов
       deck:XX     — конкретный Дек (01, 02, 03, FF и т.д.)
       boot        — stage1 / stage2
       drivers     — драйверы (ahci, xhci, vga, …)
       ipc         — notify / result pipeline
       shell       — пользовательская оболочка
       boxlib      — userspace-библиотека

   Примеры корректных сообщений:

       feat(tagfs): добавить поиск по нескольким тегам одновременно
       fix(guide): не падать при prefix_idx > MAX_PREFIXES
       refactor(scheduler): упростить вычисление SmartScore
       docs(deck:02): описать опкоды Storage Deck
       build: добавить цель make run-debug для QEMU с GDB

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

3. РАБОЧИЙ ПРОЦЕСС

   3.1. Перед началом работы — убедиться, что ветка актуальна:

       git pull origin main

   3.2. Посмотреть, что изменилось:

       git status          # список изменённых файлов
       git diff            # детальный diff несостейдженных изменений
       git diff --staged   # diff того, что уже в staged

   3.3. Добавлять файлы явно, а не через git add -A:

       git add src/kernel/core/guide/guide.c
       git add src/kernel/core/guide/guide.h

       # или добавить всё в конкретной директории:
       git add src/kernel/tagfs/

   3.4. Создать коммит:

       git commit -m "fix(guide): не входить в бесконечный цикл при пустом ring"

       # если нужно тело — открыть редактор:
       git commit

   3.5. Запушить:

       git push origin main

4. ПРОСМОТР ИСТОРИИ

   git log --oneline                   # краткий лог
   git log --oneline --graph           # граф веток
   git log --oneline src/kernel/       # история конкретной директории
   git show <hash>                     # содержимое конкретного коммита
   git blame src/kernel/main.c         # кто и когда менял каждую строку

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

5. ИСПРАВЛЕНИЕ ОШИБОК

   5.1. Отменить staged-изменения (до коммита):

       git restore --staged src/kernel/main.c

   5.2. Откатить изменения в файле (до staged):

       git restore src/kernel/main.c

   5.3. Поправить сообщение последнего коммита (если ещё не запушен):

       git commit --amend -m "правильное сообщение"

   5.4. Отменить последний коммит, сохранив изменения в файлах:

       git reset HEAD~1

   ВНИМАНИЕ: никогда не делай --amend и force-push на main.
   Если коммит уже запушен — создай новый коммит с исправлением.

6. РАБОТА С ВЕТКАМИ (для больших фич)

   # создать ветку и переключиться на неё
   git checkout -b feat/tagfs-multi-tag-search

   # после завершения работы — слить в main
   git checkout main
   git merge feat/tagfs-multi-tag-search

   # удалить ветку после слияния
   git branch -d feat/tagfs-multi-tag-search

   Ветки называются по схеме:
       feat/<область>-<краткое-описание>
       fix/<область>-<краткое-описание>

7. СБОРКА И ПРОВЕРКА ПЕРЕД КОММИТОМ

   Перед коммитом обязательно убедиться, что проект собирается:

       make clean && make && make run

   Коммитить сломанную сборку — запрещено.

8. ЧТО НЕ КОММИТИТЬ

   Следующее уже прописано в .gitignore, но помни об этом:

   - бинарники ядра (boxos.bin, boxos.iso)
   - объектные файлы (*.o, *.a — кроме заранее собранных артефактов boxlib)
   - временные файлы редакторов (*.swp, .DS_Store)
   - файлы с секретами или личными ключами

   Действие                              Команда
   ─────────────────────────────────────────────────────────────

   Diff несохранённого                   git diff
   Добавить файл                         git add <путь>
   Коммит                                git commit -m "<msg>"
   Пуш                                   git push origin main
   Пул                                   git pull origin main
   История                               git log --oneline
   Новая ветка                           git checkout -b <name>
   Отменить staged                       git restore --staged <файл>
   Откатить файл                         git restore <файл>
   Отменить последний коммит (локально)  git reset HEAD~1
