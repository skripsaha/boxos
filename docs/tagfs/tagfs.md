Название: Tagfs (Семантическая файловая система) Тип: Плоское объектное хранилище с многомерной индексацией. Философия: Файлы не «лежат» где-то, они «обладают свойствами». Поиск — это не обход дерева, а фильтрация облака.

1. Структура хранения
   Tagfs полностью отказывается от классической иерархии (директорий). На физическом уровне она состоит из двух компонентов:

Object Store: Массив сырых данных (блоков), привязанных к уникальному ID объекта.

Tag Index: База метаданных, где каждому ID объекта соответствует набор тегов.

2. Форматы тегов
   Теги в системе делятся на два типа, каждый из которых имеет свою роль в безопасности и логике.

А. Обычные теги (Basic Tags)

Используются для организации и поиска. Формат: key:value.

Примеры: type:photo, project:mars, year:2025, status:draft.

Особенности: Пользователь может создавать любые ключи и значения.

Б. Системные теги (System Tags)

Используются ядром для управления правами и состоянием. Формат: слово.

app — исполняемый файл (приложение).

utility — системная утилита с повышенными правами.

system — критический компонент ОС.

trashed — файл помечен на удаление (аналог корзины).

hidden — скрыть из обычной выдачи.

3. Механика доступа: Пересечение (Intersection)
   Доступ к файлам осуществляется через формирование запроса. Чтобы «открыть папку», вы просто указываете набор тегов.

Пример запроса: [type:doc, project:secret] Логика: Система мгновенно находит пересечение множеств ID, у которых есть оба этих тега.

Files={ID∣(type:doc∈TagsID)∧(project:secret∈TagsID)}

4. Команда use и Контекст
   Это «киллер-фича» BoxOS. Команда use настраивает линзу, через которую процесс видит файловую систему.

Как это работает: Когда ты вводишь use project:x, система автоматически добавляет этот тег ко всем последующим запросам.

Результат: Для процесса (и для тебя в интерфейсе) исчезает всё, что не относится к project:x. Система становится чистой, фокусируя тебя на задаче.

Автоматизация: При входе в виртуальную рабочую зону (например, при запуске IDE) контекст может переключаться автоматически.

5. Преимущества перед классическими ФС (NTFS/Ext4)
   Характеристика Классическая ФС (Папки) Tagfs (Теги)
   Организация Иерархическая (один файл в одной папке). Семантическая (файл в сотнях категорий сразу).
   Поиск Медленный (сканирование дерева). Мгновенный (индексный поиск).
   Дублирование Нужно делать ярлыки/ссылки. Не требуется. Теги — это и есть ссылки.
   Контекст Нет. Глубокая интеграция с планировщиком.
6. Взаимодействие с Decks
   Работа с Tagfs происходит через Storage Deck (0x02).

Процесс отправляет notify() с префиксом 02:01 (TAG_QUERY).

Storage Deck ищет в индексе подходящие ID.

Результат (список ID) возвращается в Result Page.

Для чтения данных используется префикс 02:05 (OBJ_READ).

7. Безопасность и Наследование
   Системные теги (например, system) не могут быть изменены обычным процессом без префикса от System Deck (0xFF).

При создании нового файла (через OBJ_CREATE) он автоматически наследует текущий активный контекст use. Если ты работаешь в use project:os, новый файл сразу получит этот тег.
