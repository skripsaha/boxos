# Changelog Generator

## Purpose
Produce a short, useful changelog that explains what changed and why.

## When to use
Use after:
- code changes
- refactors
- subsystem redesign
- bug fixes
- API changes
- performance work
- release prep

## Rules
- Be concise.
- Explain the reason for the change, not just the change itself.
- Mention compatibility impact if any.
- Mention risk if any.
- Do not write marketing language.
- Do not add vague filler.

## Workflow
1. Summarize the change in one sentence.
2. List the main technical changes.
3. Explain the reason.
4. Note compatibility or migration impact.
5. Note verification status.
6. Note remaining risk.
